# Application Usecase: Update

## Когда использовать

Используй `update`, когда сценарий изменяет существующую сущность или часть ее состояния.

## Что обычно делает update-usecase

- Собирает зависимости через `build(session)`.
- Получает идентификатор и изменяемые поля.
- При необходимости читает текущее состояние перед обновлением.
- Выполняет прикладные проверки и нормализацию данных.
- Передает итоговые изменения в repository.

## Базовые правила

- Используй `@sessionmaker.write`.
- Для предварительных проверок добавляй `async def validate(...)`.
- `validate(...)` вызывает только нужные repository-проверки, чаще всего `exists()`, и поднимает `AlreadyExists`, `NotFound` или другие доменные ошибки.
- `validate(...)` ничего не возвращает и принимает только аргументы, нужные для проверки.
- Проверки конфликтов и недопустимых переходов состояния должны оставаться в usecase или домене, но не в HTTP-слое.
- Не обновляй сущность вручную через таблицы или raw SQL.
- Если часть полей не меняется, не изобретай новый формат patch, если рядом уже есть принятый способ.

## Ориентир

Если в домене нет готового `update.py`, бери ближайший write-usecase того же домена и сохраняй его форму.
