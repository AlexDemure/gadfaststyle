# Application Usecase: Search

## Когда использовать

Используй `search`, когда сценарий возвращает коллекцию данных по `filters`, `sorting` и при необходимости `pagination`.

## Что обычно делает search-usecase

- Собирает зависимости через `build(session)`.
- Принимает уже подготовленные параметры поиска из HTTP-схемы.
- Без дополнительной трансформации или с минимальной orchestration-логикой прокидывает `filters`, `sorting`, `pagination` как словари в repository adapter.
- Возвращает `tuple[list[Model], int]`, если есть пагинация, или просто список для легких справочников.

## Базовые правила

- Используй `@sessionmaker.read`.
- Не дублируй в usecase разбор `filters`, `sorting`, `pagination`, если они уже подготовлены схемой.
- Не собирай в usecase builder-объекты `Filter`, `Sorting`, `Pagination` для бизнес-ручек `:search`.
- Не строй SQLAlchemy statement в usecase.
- Для `:search` основная логика поиска должна собираться в repository adapter и entity-specific CRUD `search(...)`.
- `paginated(...)` используй для отдельных ручек `:paginated`, когда внешнему сервису нужна простая пакетная выгрузка без бизнес-логики поиска.
- Пост-обработку результата, если она нужна, делай отдельным шагом перед `return`, а не внутри выражения возврата.
- Для новых сценариев не используй `list` как паттерн выборки, даже если legacy-пример в проекте еще существует.

## Эталон

Ориентируйся на search-сценарии текущего домена. Если в кодовой базе есть только legacy `list`, используй его как источник общей формы `read`-usecase, но не копируй название операции.
