## Процесс

Выполни синхронизацию `.specs/` с фактическим состоянием проекта:

1. Прочитать релевантные разделы из `.instructions/`.
2. Просканировать `src/`, `tests/` и `.specs/`.
3. Сопоставить каждый файл проекта с соответствующим `spec.md`.
4. Найти структурные, файловые и содержательные расхождения.
5. Создать, обновить, удалить или переместить `spec.md` так, чтобы `.specs/` снова зеркалила проект.
6. Сохранить один runtime-файл в `.claude/runtime/sync/<YYYY-MM-DD_HH-MM-SS>_<slug>.md`.

## Требования

- `.specs` должна повторять `src/` и `tests/` 1:1 по файлам.
- Один файл проекта должен соответствовать одному `spec.md`.
- Путь `src/infrastructure/databases/postgres/migrations/` исключается из синхронизации и не должен документироваться в `.specs/`.
- Синхронизация не меняет `src/` и `tests/`, она меняет только `.specs/`.
- Если содержание файла не удается уверенно восстановить в spec, это должно быть зафиксировано как проблема синхронизации.

## Формат runtime-файла

Runtime-файл должен начинаться со строки `Синхронизация: YYYY-MM-DD HH:MM:SS`.

Runtime-файл ведется как последовательность шагов синхронизации. Каждый этап оформляй отдельным заголовком с номером шага. Формат заголовка: `## №<n> Синхронизатор спецификаций - <этап>`.

Runtime-файл должен показывать полное содержимое измененных `spec.md`. Для каждого `spec.md` используй заголовок в формате `edited:<path>`, `created:<path>`, `moved:<old> -> <new>`, `deleted:<path>`. Для `edited`, `created` и `moved` показывай полную актуальную версию `spec.md`. Внутри содержимого помечай изменения строками `+` и `-`, если это помогает быстро увидеть правки. Для `deleted` достаточно только заголовка.
