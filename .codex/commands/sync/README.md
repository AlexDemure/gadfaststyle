## Назначение

`sync` описывает синхронизацию `.specs/` с фактическим состоянием проекта.

## Процесс

1. Прочитать релевантные разделы из `.instructions/`.
2. Просканировать `src/`, `tests/` и `.specs/`.
3. Сопоставить каждый файл проекта с соответствующим `spec.md`.
4. Найти структурные, файловые и содержательные расхождения.
5. Создать, обновить, удалить или переместить `spec.md` так, чтобы `.specs/` снова зеркалила проект.
6. Сохранить один runtime-файл в `.codex/runtime/sync/<YYYY-MM-DD_HH-MM-SS>_<slug>.md`.

## Требования

- `.specs` должна повторять `src/` и `tests/` 1:1 по файлам.
- Один файл проекта должен соответствовать одному `spec.md`.
- Путь `src/infrastructure/databases/postgres/migrations/` исключается из синхронизации и не должен документироваться в `.specs/`.
- `spec-sync` не меняет `src/` и `tests/`, он меняет только `.specs/`.
- Если содержание файла не удается уверенно восстановить в spec, это должно быть зафиксировано как проблема синхронизации.

## Результат

- Runtime-файл должен начинаться со строки `Синхронизация: YYYY-MM-DD HH:MM:SS`.
- Runtime-файл ведется как последовательность шагов синхронизации.
- Каждый этап оформляй отдельным заголовком с номером шага.
- Формат заголовка: `## №<n> Синхронизатор спецификаций - <этап>`.
- Runtime-файл команды должен показывать полное содержимое измененных `spec.md`.
- Для каждого `spec.md` используй заголовок в формате `edited:<path>`, `created:<path>`, `moved:<old> -> <new>`, `deleted:<path>`.
- Для `edited`, `created` и `moved` показывай полную актуальную версию `spec.md`.
- Внутри содержимого помечай изменения строками `+` и `-`, если это помогает быстро увидеть правки.
- Для `deleted` достаточно только заголовка.

Синхронизация: 2026-04-01 14:20:35

## №1 Синхронизатор спецификаций - Сканирование

- просканированы `src/`, `tests/` и `.specs/`

## №2 Синхронизатор спецификаций - Обновление spec-файлов

## Назначение

Поднять HTTP-сервер приложения и запустить ASGI runtime.

## Поведение

Модуль создает приложение, настраивает жизненный цикл и запускает uvicorn с параметрами из конфигурации.

## Входы

- Конфигурация приложения.
- Factory функции для сборки `FastAPI`.

## Выходы

- Запущенный ASGI-сервер.

## Зависимости

- `uvicorn`
- `FastAPI`
- Модуль bootstrap-конфигурации

## Ограничения

- Порт и host берутся только из конфигурации.
- Модуль не содержит бизнес-логики.
