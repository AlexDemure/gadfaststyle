## Назначение

`code` описывает полный цикл разработки по бизнес-задаче.

## Процесс

1. Все агенты читают релевантные разделы из `.instructions/` и соблюдают правила проекта.
2. `system-analyst` анализирует задачу и создает, обновляет, удаляет или перемещает `spec.md` в `.specs/`.
3. `system-analyst` проставляет даты и время в измененных `spec.md`.
4. `backend` реализует код в `src/` на основе соответствующих `spec.md`.
5. `tester` реализует тесты в `tests/` на основе соответствующих `spec.md`.
6. `backend` и `tester` при своих изменениях обновляют даты и время в соответствующих `spec.md`.
7. `team-lead` проверяет результат работы в `src/` и `tests/` на основе `spec.md`, проводит code review и дает обратную связь.
8. Если нужны правки, `backend` и `tester` повторно вносят изменения и команда продолжает дописывать этот же runtime-файл.
9. Если правки не нужны, `team-lead` подводит итог работы.
10. Команда сохраняет один runtime-файл в `.codex/runtime/code/<YYYY-MM-DD_HH-MM-SS>_<slug>.md`.

## Требования

- Один путь проекта соответствует одному `spec.md`.
- `backend` работает только в `src/`.
- `tester` работает только в `tests/`.
- `team-lead` проверяет результат по `spec.md`, проектным инструкциям и фактическим изменениям в `src/` и `tests/`.
- Если по задаче требуется несколько итераций, весь процесс дописывается в один и тот же runtime-файл.
- Во всех измененных `spec.md` должны быть проставлены актуальные дата и время с секундами.

## Результат

- Runtime-файл должен начинаться со строки `Разработка: YYYY-MM-DD HH:MM:SS`.
- Runtime-файл ведется последовательно и дополняется по ходу работы агентов.
- Каждый этап оформляй отдельным заголовком с номером шага.
- Формат заголовка: `## №<n> <роль> - <этап>`.
- `system-analyst` добавляет в runtime-файл полное содержимое затронутых `spec.md`.
- Для каждого `spec.md` используй заголовок в формате `edited:<path>`, `created:<path>`, `moved:<old> -> <new>`, `deleted:<path>`.
- Для `edited`, `created` и `moved` показывай полную актуальную версию `spec.md`.
- Для `deleted` достаточно только заголовка.
- После блока аналитика добавляй блоки работы `backend`, `tester` и `team-lead` в порядке выполнения.
- `team-lead` в конце каждой итерации фиксирует либо замечания на доработку, либо текстовый итог работы.

## Назначение

Обновить аккаунт текущего пользователя.

## Поведение

Usecase принимает изменяемые поля аккаунта, валидирует их и сохраняет новое состояние.

## Входы

- `account_id`
- `external_id`

## Выходы

## Зависимости

- Repository adapter аккаунтов.
- Security-утилиты для шифрования полей.

## Ограничения

- Обновляются только поля из контракта сценария.
- Конфликты проверяются до записи.
