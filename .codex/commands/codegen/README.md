## Назначение

`codegen` описывает полный цикл генерации кода по бизнес-задаче.

## Workflow

1. Прочитать релевантные разделы из `.instructions/`.
2. `system-analyst` анализирует задачу и генерирует spec prompt-ы в `.specs/tasks/<task-slug>/`.
3. `backend` реализует production-код по prompt-ам и стилю проекта.
4. `tester` реализует тесты по prompt-ам и стилю проекта.
5. `team-lead` проверяет результат, фиксирует замечания или собирает финальный отчет.
6. Все промежуточные и итоговые артефакты сохраняются в `.codex/tasks/<YYYY-MM-DD>_<slug>/`.

## Требования

- Перегенерация должна быть идемпотентной, если spec prompt не менялся.
- Если код вручную изменили после генерации, агент должен зафиксировать drift.
- Prompt-ы должны быть мелкими и управляемыми: endpoint, usecase, schema, тестовый набор.
- Финальный отчет должен ссылаться на использованные spec prompt-ы и измененные файлы.

## Артефакты работы

### System Analyst

- Короткое техническое описание задачи
- Блок `Specs:`
- В блоке `Specs:` список файлов в `.specs/` в формате `- <action>: <path>`
- Допустимые `action`: `create`, `update`, `delete`, `move`
- После выполнения обновить в затронутых `spec.md` поля `created_at` или `system_analyst_updated_at`

### Backend

- `Созданные файлы:` со списком путей в `src/`
- `Измененные файлы:` со списком путей в `src/`
- `Удаленные файлы:` со списком путей в `src/`
- `Перемещенные файлы:` со списком путей в формате `<old> -> <new>`
- `Сводка реализации:` короткое описание внесенных изменений
- `Diff:` фрагменты кода, показывающие что изменилось
- `Проверки:` список выполненных проверок
- `Блокеры:` список блокеров, если они есть
- `Drift:` список расхождений, если они есть
- После выполнения обновить в затронутых `spec.md` поле `backend_synced_at`

### Tester

- `Созданные тесты:` со списком путей в `tests/`
- `Измененные тесты:` со списком путей в `tests/`
- `Удаленные тесты:` со списком путей в `tests/`
- `Перемещенные тесты:` со списком путей в формате `<old> -> <new>`
- `Сводка покрытия:` короткое описание покрытых сценариев
- `Проверки:` список выполненных проверок
- `Пробелы:` список непокрытых сценариев, если они есть
- `Drift:` список расхождений, если они есть
- После выполнения обновить в затронутых `spec.md` поле `tester_synced_at`

### Spec Sync

- `Структурные расхождения:` список расхождений по каталогам и структуре
- `Файловые расхождения:` список отсутствующих, лишних и устаревших spec-файлов
- `Содержательные расхождения:` список расхождений между кодом и содержимым `spec.md`
- `Обновленные Specs:` список созданных и обновленных `spec.md`
- `Проблемы синхронизации:` список проблем, если они есть
- После выполнения обновить в затронутых `spec.md` поле `spec_sync_synced_at`

### Team Lead

- Итоговый статус: `approved` или `changes_required`
- Список замечаний в формате `- <severity>: <text>`
- Если статус `changes_required`, список требуемых исправлений в формате `- fix: <text>`
- Если статус `approved`, финальный отчет по задаче
- После приемки обновить в затронутых `spec.md` поле `team_lead_synced_at`
